TDSQL For MySQL认证-单选题

  1. TDSQL(mysql)下列关于主键错误的是().D

    A. InnoDB表必须指定主键
    B. 尽量游免更新主键字段
    C. 如果存在多个唯一健，考虑最常用的唯一键作为主键
    D. 主键值可以为空
    在 TDSQL（MySQL）中，主键是用于唯一标识表中每条记录的一列或一组列。主键具有以下特点：

    主键必须是唯一的，不能重复。
    主键不能为 NULL。
    主键可以由一列或多列组成。
    主键可以用来引用其他表中的记录。
    因此，选项 D 不正确，主键值不可以为空。如果一个记录的主键为空，则该记录将不会被插入到表中，或者在更新主键字段时会引发错误。

    选项 A 是正确的，如果表使用 InnoDB 引擎，则必须指定主键，因为 InnoDB 表必须有聚集索引，而主键可以作为聚集索引。

    选项 B 也是正确的，因为更新主键字段可能会引发性能问题。这是因为主键上的索引通常是唯一索引，每次更新都需要先检查是否有重复的键值。

    选项 C 可能是正确的，但具体取决于数据的实际情况。通常情况下，最好选择一个唯一且经常用于查询的键作为主键。

  2. 在mysql 中innodb引擎表T1，执行update t1 set a=2 where id =1；语句，id列有索引，下面描述正确的是（）**C**
    A. 加表锁
    B. 不加锁
    C. 加行锁
    D. 加字段锁
    在 MySQL 中，InnoDB 存储引擎使用行级锁来保护数据的并发访问。当执行 UPDATE 语句时，InnoDB 引擎会自动对涉及到的行加上行级锁，以保证在更新过程中不会发生数据冲突。

    在这个例子中，执行 UPDATE T1 SET A=2 WHERE ID=1 语句时，InnoDB 引擎会自动对 ID=1 这一行加上行级锁，防止其他并发事务对这一行进行修改或删除。需要注意的是，如果表 T1 中有其他列上的索引，InnoDB 引擎也会自动对这些索引加锁，以保证数据的一致性。

    选项 A 不正确，因为在 InnoDB 引擎中，一般不会使用表级锁，而是使用行级锁。

    选项 B 不正确，因为 InnoDB 引擎会自动对涉及到的行加上行级锁，以保证数据的一致性。

    选项 D 不正确，因为在 InnoDB 引擎中，没有字段级别的锁，只有行级别和表级别的锁。

  3. 负责mysqld 进程存活检查并进行拉起的进程是（）.A
    A. Oc_agent
    B. Scheduer.
    C. manager
    D. hdfs
    在 TiDB 数据库中，负责 MySQL 实例存活检查并进行拉起的进程是 TiUP 的 Oc agent。Oc agent 是一个轻量级的进程，它负责监控 TiDB 集群的各个组件（包括 TiDB、TiKV、TiFlash、PD 等），并在组件异常或宕机时及时进行处理，以保证集群的高可用性和可靠性。

    Oc agent 会通过心跳检查来判断 MySQL 实例是否存活，如果发现 MySQL 实例异常或宕机，则会尝试将其拉起来。Oc agent 还可以通过配置文件来控制拉起 MySQL 实例的方式和参数，以适应不同的场景和需求。

    选项 B、C、D 不正确，因为这些进程与 TiDB 数据库的存活检查和拉起过程无关。Scheduler 是调度器，用于调度 TiDB 集群中各个组件的工作任务；Manager 是集群管理器，用于管理 TiDB 集群的状态和配置信息；HDFS 是分布式文件系统，与 TiDB 数据库无直接关系。

  4. 使用mysql客户端透传登方式录后，可以查看库中有哪些表是分表的语句是（）.A
    A. /*proxy*/show table with shardkey;
    B. show shardkey tables;
    C. /*proxy*/show tables;
    D. /*proxy*/show shardkey tables;
    在 TiDB 数据库中，如果使用了分表技术，可以通过 show table with shardkey; 命令来查看表的分片情况。这个命令可以在 MySQL 客户端中执行，也可以在 TiDB Dashboard 中执行。

    需要注意的是，在使用 MySQL 客户端透传方式登录 TiDB 数据库时，需要在 SQL 语句前面添加 /proxy/ 注释，以便让 TiDB Proxy 知道这个 SQL 语句需要透传到后端的 TiDB 节点上执行。因此正确的命令应该是 /proxy/show table with shardkey;。

    选项 B、C、D 都不正确。show shardkey tables;、show tables; 和 show shardkey tables; 这些命令都不能正确查看表的分片情况。

  5. TDSQL 修改字段类型命令正确的是（）.B

    B. alter table t1 modify e varchar(30);

    D. alter table t1.e alter varchar(30);
    在 TDSQL 中，要修改表的字段类型，可以使用 alter table 命令，并指定要修改的字段名称和新的字段类型。例如，将表 t1 中的字段 e 的类型修改为 varchar(30)，正确的命令是 alter table t1 modify e varchar(30);。

    选项 A、C、D 都不正确。alter table t1 alter e varchar(20); 和 alter table t1.e modify varchar(30); 都缺少了要修改的字段的关键词 modify；alter table t1.e alter varchar(30); 则是语法错误的命令。

  6. 使用mysql客户端登录TDSQL时，加哪个参数可以使用透传功能?C
    A. -a
    B. -b
    C. -c
    D. -d
    在使用 MySQL 客户端登录 TDSQL 数据库时，可以使用 -c 参数来开启透传功能。透传功能可以让 MySQL 客户端向后端的 TDSQL 节点发送原生的 MySQL 协议报文，可以使用原生的 MySQL 命令和语法，包括使用 TiDB 不支持的 MySQL 特性和语法。使用透传功能时，需要在 SQL 语句前面添加 /+/ 注释，以便让 TDSQL Proxy 知道这个 SQL 语句需要透传到后端的 TDSQL 节点上执行。例如：

    mysql -h host -P port -u user -p -c

    mysql> /+/show variables like ‘%version%’;
    选项 A、B、D 都不正确。-a、-b 和 -d 都不是 MySQL 客户端支持的参数。

  7. 当检测到死锁时，TDSQL InnoDB 存储引擎如何处理它们?D
    A. 两个受影响的事务都将回滚。
    B. 受职响的事务等待inodb lock wait 超时，然后回滚。
    C. 一个受影响的事务将回滚，另一个允许继续.
    D. 事务隔离级别决定回滚哪个事务.
    当 TDSQL 检测到死锁时，InnoDB 存储引擎将根据事务的隔离级别来处理死锁。InnoDB 支持四种隔离级别：READ UNCOMMITTED、READ COMMITTED、REPEATABLE READ 和 SERIALIZABLE。

    READ UNCOMMITTED：在该隔离级别下，TDSQL 不会发生死锁。
    READ COMMITTED：InnoDB 选择优先级更高的事务来回滚，低优先级的事务将继续执行。
    REPEATABLE READ：InnoDB 选择最先发现死锁的事务来回滚，另一个事务将继续执行。
    SERIALIZABLE：在该隔离级别下，TDSQL 不会发生死锁。
    因此，在 TDSQL 中，死锁的处理方式是根据事务的隔离级别来决定哪个事务将被回滚。

  9. TDSQL（MySQL版）关于强同步策略描述正确的是?C
    A. 主机不等从机应答直接返客户端成功
    B. 主机等待多数从机应答再返回客户端成功
    C. 主机等待至少一台从机应答成功后才返回客户端成功
    D. 主机等待所有从机应答成功后才返回客户端成功
    TDSQL (MySQL版) 采用的是基于 Paxos 协议的强同步策略。当主机执行完事务后，必须等待至少一台从机的应答成功后才能返回客户端成功。这样做是为了保证数据在主从复制中的强一致性。在某些场景下，如果从机同步异常，主机会阻塞事务，等待从机同步成功后才能返回客户端。

  15. TDSQL（MySQL版）分布式实例中哪些表只存在第一个set上?A

  17. 关于TDSQL 广播表说法正确的是（）?D
    A. 广播表可以将表数据均匀拆分到各个set
    B. 广播表一般适合数据量较大的表
    C. 广播表适合用于频繁更新的表
    D. 广播表适用于更新较少的小表
    广播表适用于更新较少的小表，因为在广播表中，数据是完全复制到每个 set 的，这样对于数据量较大的表会占用过多的存储空间，对于频繁更新的表会增加同步的负担。因此，广播表一般适用于数据量较小、更新较少的场景，如配置表等。

    MySQL报错error 1062 Duplicate entry ‘…’ for key ‘PRIMARY’表示字段值重复，入库失败。这通常是由于尝试向已有主键的列中插入重复的值导致的。

  35. TDSQ(MYSQL)分布式数据库查看某表数据，使用武力透传法/* set set_xxx*/select * from table 在不同分片查得数据相同，可能因为？**B**
    A. 分片键选择有问题
    B. 命令行MySQL链接没有使用-c参数解析注释
    C. 此表为单表
    D. 偶然事件各个分片数据相同
    使用“武力透传”注释时，应该在命令行中使用-c参数，将注释传递给后端存储节点。如果不使用-c参数，则注释将被客户端应用程序截获，不会被传递到存储节点。因此，不同分片的数据相同可能只是因为客户端应用程序拦截了注释，而没有将其传递到存储节点。

  44. 下列哪个命令不属于DML？**C**
    A. DELETE
    B. UPDATE
    C. ALTER
    D. INSERT
    C. ALTER命令不属于DML，它是DDL（数据定义语言）的一种，用于修改数据库对象的结构。DELETE、UPDATE和INSERT是DML（数据操作语言）的一种，用于操作数据库中的数据。

  45. TDSQ(MYSQL) 下列三种删除数据的命令，那种会删除表结构？****B****
    A. delete
    B. Drop
    C. Tun
    Drop 命令用于删除整个表，包括表结构和数据；delete 命令用于删除表中的数据，保留表结构；Truncate 命令也是用于删除表中的数据，但是会将表恢复到初始状态。

  48. 关于TDSQ(MYSQL) 备份的意义，说法错误的是？ A

    C. 历史数据分析，直接在生产实例上分析对线上业务造成冲击
    D. 多重灾难，导致主备数据全部损毁，可以恢复数据
    备份的主要作用是在数据出现问题时能够快速恢复数据，防止数据丢失或不可恢复。备份数据通常用于误删数据、数据恢复、历史数据分析等场景。其中，备份数据可用于重建备机，但并不是备份的主要意义。
